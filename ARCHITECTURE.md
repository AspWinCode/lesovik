## 0. Главное (TL;DR)

- **Тип**: Green-field MVP корпоративной No-Code платформы. Аналоги: Creatio Studio, Bubble, Retool, OutSystems — но on-prem и в РФ (152-ФЗ).
- **Самые рискованные узлы**, на которых проект чаще всего «ломается» — закладываем правильно сразу:
  1. **Динамические сущности**. Если выбрать неверно (EAV или dynamic DDL) — производительность 1М записей не вытянем. Решение: **PostgreSQL + JSONB + GIN-индексы + материализованные представления** под горячие запросы.
  2. **Rules Engine с тайм-аутом 113 сек**. Это пользовательский код. Если запускать в основном процессе — DoS и SQL-инъекции. Решение: **выделенный sandbox-воркер (Celery + изолированный процесс), безопасный DSL (без `eval`), параметризованные построители запросов**.
  3. **Workflow Engine с SLA и эскалацией**. Самописный — растянет 2 спринта на отладку гонок. Решение: **state machine поверх Celery beat + outbox-таблица для надёжной доставки уведомлений**.
  4. **Поле-уровневые права + ABAC**. Самая частая дыра в no-code (IDOR через API). Решение: **единый PolicyEngine, который обязан прогоняться для каждого CRUD; запрет «голого» SQL мимо него**.
- **Срок реалистичен**: ТЗ хорошо нарезано на 6 спринтов. Но MVP получится «работающим минимумом», а не «production-grade на 500 RPS» — нагрузочное и инфобез-приёмка в спринте 6 могут потребовать ещё 2 недели на устранение замечаний (заложено как риск).
- **Что доделано в плане ТЗ**: добавлен **Спринт 0** (инфраструктура и threat-model), **2FA перенесена из 5-го спринта в 1-й** (она часть аутентификации), в каждый спринт добавлена «фоновая дорожка» инфобеза и нагрузочный smoke-тест.

## 3. Ключевые архитектурные решения (где можно «сломать» проект)

### 3.1 Хранение динамических сущностей — JSONB + системная метамодель

```
metamodel.entity (id, code, name, module_id, schema_json, created_at,…)
metamodel.field   (id, entity_id, code, type, required, ref_entity_id, formula, validators_json,…)
metamodel.relation(id, entity_id_from, entity_id_to, kind, on_delete,…)

data.record  (
   id UUID PK,
   entity_id UUID FK,
   tenant_id UUID,                -- задел на multi-tenant
   payload JSONB,                  -- пользовательские поля
   status VARCHAR,                 -- системное
   is_deleted BOOLEAN, deleted_at, deleted_by, created_by, created_at, updated_at, version INT
)
-- индексы:
-- (entity_id, is_deleted) BTREE
-- GIN на payload (jsonb_path_ops)
-- частичные индексы на горячие поля: ((payload->>'status'))
-- партиционирование data.record по entity_id (hash) при росте >10M
```

Почему не EAV: на 1М записей JOIN по EAV — катастрофа. Почему не dynamic CREATE TABLE: миграции, alembic, бэкапы и реплика начнут сходить с ума при сотнях пользовательских сущностей. **JSONB — лучший компромисс**: гибкость + индексы + честный SQL.

Цена: запросы строит наш QueryBuilder поверх SQLAlchemy, **никаких f-string SQL**. См. инфобез.

### 3.2 Rules Engine — изолированный sandbox

- **DSL, не Python**: правила сериализуются в JSON-AST (`{op: "and", args: [...]}`). Никакого `eval` / `exec`.
- Интерпретатор написан на Python, читает AST, дёргает безопасные функции (`SUM`, `IF`, `LOOKUP(entity, id)`).
- LOOKUP / запись данных — через **тот же сервисный слой**, что и API. Это автоматически прогоняет PermissionService и аудит. Никаких прямых SQL.
- Тайм-аут 113 сек: исполнение — отдельный Celery task с `time_limit=113`, `soft_time_limit=110`. При превышении — `SoftTimeLimitExceeded` → транзакция откатывается, в журнал пишется ошибка.
- Защита от циклов: при компиляции правил строим граф зависимостей `field → field`; перед публикацией прогоняем DFS, циклы блокируем (ТЗ 3.5.3).
- Конфликты правил (ТЗ 3.5.4): на одно поле — одно правило валидации/автозаполнения; при попытке создать второе UI блокирует. Конфликты по приоритету фиксируются в `rule_conflict_log`.

### 3.3 Workflow Engine — state machine + outbox

- Метамодель процесса: `process_definition (стадии, переходы, SLA, эскалация)`.
- Экземпляр: `process_instance (entity_record_id, current_stage, history_json, started_at, finished_at, status)`.
- Переходы — атомарно в одной транзакции, обновление `process_instance.history` + запись в `audit_log`.
- SLA-таймеры: `process_instance_timer (process_id, fires_at, kind: escalation_l1|l2|auto_change_assignee)` + Celery beat сканирует каждую минуту. Альтернатива (вторая очередь): **Temporal** для надёжности.
- Уведомления отправляются **через outbox-таблицу** (`notification_outbox`) — гарантия доставки: транзакция БД пишет outbox-строку → отдельный воркер шлёт → ретраи с экспонентой → DLQ.

### 3.4 Модульная архитектура — плагин-контракт

- Каждый модуль — **папка `modules/<name>/`** в репозитории + **manifest.yaml**: `version`, `depends_on`, `entities`, `pages`, `rules`, `permissions`.
- Установка модуля = выполнение его миграций Alembic + регистрация метамодели в `metamodel.module`.
- Совместимость: SemVer; при обновлении — `migration.py` модуля. До-коммит-проверка: `pytest tests/modules/<name>/` запускает интеграционные тесты на чистой БД.
- Конфликты (ТЗ 2.5.2): уникальные коды сущностей `<module_code>.<entity_code>`, переименование при коллизии через мастер установки.

### 3.5 Совместное редактирование — пессимистическая блокировка

ТЗ требует именно «блокировки редактора» (3.2.1), это упрощает MVP (не нужен CRDT). Реализация: Redis-ключ `editor_lock:app:<id>` с TTL 5 мин, продление каждые 30 сек heartbeat. Принудительный сброс — отдельный endpoint администратора.

### 3.6 Версионирование приложений

`application_version (app_id, version_no, snapshot_json, published_at, published_by)`. Snapshot — полный JSON метамодели приложения (страницы, блоки, правила). Откат = создание новой версии из snapshot прошлой. Никогда не редактируем опубликованную версию in-place.

## 4. Масштабируемость

| Размерность | Подход в MVP | Запас на рост |
|---|---|---|
| Backend | 4 FastAPI-инстанса за Nginx, sticky-sessions не нужны (JWT) | Горизонтально без ограничений, `gunicorn -w` + переезд в K8s |
| БД | Managed PostgreSQL primary + 1 read-replica | Партиционирование `data.record` по `entity_id`; шардирование Citus при необходимости |
| Очереди | Redis 7 single-node + Celery (4 воркера) | Redis Sentinel/Cluster; разделение очередей по приоритету (`fast`, `slow`, `sandbox`) |
| Файлы | Managed S3 / MinIO 2 узла, EC-режим | MinIO Distributed (4+ узла) |
| Кэш | Redis для метамодели и прав (TTL 60 сек, инвалидация по событию) | CDN для публичной статики; pgBouncer перед PG |
| Поиск | PG FTS + GIN | OpenSearch для cross-entity и тяжёлой аналитики |
| Сетевые границы | Один VLAN, mTLS для внутреннего трафика админских API | Изоляция в K8s NetworkPolicy |

**Нагрузочные цели по ТЗ**: 500 одновременных пользователей, 1М записей в сущности, API ≤500 мс p99, отчёт ≤30 сек на 100К строк. Эти цифры закладываем в Locust-сценарии **с первого спринта** (smoke), полный нагрузочный — в спринте 6. Реальная стартовая нагрузка после go-live (50–200 одновременных) — оптимизируем под 200, потолок 500 — задельный.

## 5. Интеграции (паттерны, не «костыли»)

| Система | Очередь | Паттерн |
|---|---|---|
| Email (SMTP/IMAP) | MVP | Outbox + Celery `aiosmtplib`; IMAP poll каждые 5 мин для входящих документов |
| Excel/CSV/JSON импорт-экспорт | MVP | Стрим-парсинг (`openpyxl read_only`), маппинг колонок, валидация перед commit, отчёт |
| 1С: Предприятие | Очередь 2 | OData/HTTP-сервисы 1С → коннектор-сервис в нашем периметре, очередь обмена через outbox |
| Telegram Bot | Очередь 2 | Long polling-воркер; шаблоны с подстановкой; rate-limit Telegram API учитываем |
| Банк MT940 | Очередь 2 | Файловый импорт через ту же ImportExportService, парсер MT940 — отдельный модуль |
| Webhook (исходящий) | Очередь 2 | Outbox + ретраи с HMAC-подписью, idempotency-key |
| ЭДО (Диадок/СБИС), ЭП | Очередь 3 | Не в этом квартале |

**Сквозной принцип**: все интеграции — через сервисный слой (`IntegrationService`), а не из роутеров. Каждая интеграция изолирована за фасадом — заменить провайдера должно стоить одного класса.

REST API платформы (ТЗ 3.14.1): **тот же, что и UI использует**. Один API, два потребителя — иначе разъедутся. Аутентификация для внешних — API-ключ + scopes.

## 6. Информационная безопасность (фокус-зона)

### 6.1 Базовая модель угроз (STRIDE-по-крупному)

| Угроза | Где встречается | Что закладываем |
|---|---|---|
| **SQL-инъекция** | Динамические фильтры таблиц, Rules Engine, импорт, поиск | (1) Только параметризованные запросы через SQLAlchemy; (2) запрет `text()` без `bindparam`; (3) валидация имён полей/сущностей по whitelist из метамодели; (4) Rules Engine не строит SQL — он зовёт сервисы; (5) автотесты с sqlmap в CI staging |
| **XSS** | Rich-text блоки, KB-статьи, имена сущностей в UI | (1) React эскейпит по умолчанию; (2) для HTML-контента — DOMPurify; (3) CSP `default-src 'self'`, `script-src 'self' 'nonce-...'` |
| **CSRF** | Формы конструктора, импорт | SameSite=Lax cookies + CSRF-токен на изменяющие запросы (FastAPI middleware) |
| **IDOR / Broken access control** | Любой CRUD пользовательских записей | PolicyEngine **обязателен** на каждом обращении; RLS в PG как страховка (`tenant_id = current_setting('app.tenant_id')`) |
| **Privilege escalation** | RBAC + ABAC | Гранулярные права `app.page.block.field.record`; запрет пользователю менять свои роли; журнал аудита прав |
| **Insecure file upload** | Загрузка файлов | (1) MIME через libmagic, не Content-Type; (2) ClamAV проверка перед коммитом; (3) хранение вне docroot; (4) выдача через presigned URL с экспирацией 15 мин; (5) запрет HTML/SVG как inline (только `Content-Disposition: attachment`) |
| **SSRF** | Webhook, OAuth-redirect, импорт по URL | Whitelist доменов; запрет приватных сетей (`10/8`, `192.168/16`, `127/8`, link-local, `0.0.0.0`); DNS pinning |
| **Brute force / Credential stuffing** | Логин | Rate-limit `slowapi` (5 попыток/мин/IP + 10/час/логин); CAPTCHA после 3-х неудач; lockout 15 мин; **уведомление администратора** (ТЗ 3.13) |
| **Session fixation** | JWT-сессии | Refresh-токены с ротацией, привязка к device fingerprint; `revoke_token`-таблица; тайм-аут 30 мин (ТЗ 3.1.3) |
| **Data exfiltration через экспорт** | Экспорт XLSX/PDF | Маскирование на уровне поля (ТЗ 3.13); квоты на экспорт; уведомление о «нетипичном экспорте» (ТЗ 3.13) |
| **Кража секретов** | `.env`, ключи API | HashiCorp Vault (или Doppler в MVP); запрет коммитов через `gitleaks` в pre-commit и CI; ротация ключей раз в 90 дней |
| **Supply chain** | npm/pip пакеты | `pip-audit`, `npm audit`, **Trivy** для контейнеров; пиннинг версий; SBOM (`syft`) |
| **Несанкционированный доступ к Rules Engine** | Пользовательский DSL | Sandbox-воркер с `RLIMIT_CPU`, без сети (`unshare -n`), без файловой системы (read-only rootfs); тайм-аут 113 сек; квота памяти 512 МБ |

### 6.2 Соответствие 152-ФЗ

- Серверы и бэкапы — в РФ (требование ТЗ 3.13).
- Согласие на обработку ПДн — экран при первом входе для приглашённых внешних пользователей.
- Журнал обработки ПДн (отдельная append-only таблица `pii_access_log`), хранение 3 года.
- Маскирование номеров паспортов/телефонов/email в логах и экспортах по умолчанию.
- Право на забвение: процедура «полного удаления» данных пользователя — отдельный административный сценарий с двойным подтверждением (физическое удаление, обход soft-delete политики; единственный случай, см. ТЗ 3.9.2).
- Шифрование в покое — AES-256 на уровне диска (LUKS) + TDE для PG (через `pgcrypto` для критичных колонок: пароли, токены 2FA, ключи API).
- TLS 1.2+, HSTS, OCSP-stapling.

### 6.3 Аудит-журнал — append-only по-настоящему

- Отдельная таблица `audit_log` с `INSERT only`, **триггер на `BEFORE UPDATE/DELETE` который кидает ошибку**.
- Дополнительно: ежедневный экспорт хэш-цепочки (`prev_hash → row → hash`) в WORM-хранилище (S3 Object Lock).
- Что пишется: вход/выход, изменение прав, импорт/экспорт, CRUD по записям (id + дельта), запуск/останов процессов, конфликты правил, подозрительная активность.

### 6.4 Уведомления администратора о подозрительной активности (ТЗ 3.13)

Конкретные правила:
- ≥5 неудачных логинов за 10 мин по одному логину → email + in-app.
- Удаление >100 записей за минуту одним пользователем → блокировка операции + уведомление.
- Экспорт >10К записей за час → подтверждение и уведомление.
- Логин из новой страны / нового user-agent → уведомление пользователю и админу.

## 7. Стратегия тестирования (включая защиту от SQLi)

| Уровень | Инструмент | Что покрывает | Когда |
|---|---|---|---|
| Unit | pytest | Сервисы, валидаторы, Rules DSL-интерпретатор | Каждый PR |
| Integration | pytest + testcontainers-postgres | Сервис ↔ БД, миграции модулей, RLS | Каждый PR |
| API contract | Schemathesis (читает OpenAPI) | Фаззинг параметров, поиск 500/нарушений схемы | Каждый PR |
| E2E | Playwright + pytest-playwright | Сценарии конструктора и runtime, мобильный вид | Pre-merge + nightly |
| Frontend unit | Vitest + React Testing Library | Компоненты, хуки | Каждый PR |
| Нагрузочные | Locust | 500 одновременных, 1М записей, p99 ≤500 мс | Спринты 3, 5, 6 |
| **SAST** | Bandit (Python), semgrep (custom rules) | Поиск `eval`, `text(f"...")`, опасных API | Каждый PR |
| **SCA** | pip-audit, npm audit, OSV-scanner | Уязвимости в зависимостях | Nightly |
| **Container scan** | Trivy | CVE в образах | Каждый build |
| **Secrets scan** | gitleaks | Утечки ключей в коммитах | Pre-commit + CI |
| **DAST** | OWASP ZAP (baseline и full scan) | Активная проверка staging | Nightly |
| **SQLi-фаззинг** | sqlmap | Прицельная проверка всех изменяющих endpoint'ов | Pre-release |
| **Авторизация** | пакет custom pytest-фикстур | Каждая роль × каждый endpoint = матрица; проверка IDOR | Каждый PR |
| **Penetration test** | внешний подрядчик | Полный pentest до go-live | Спринт 6 |

Покрытие — целевое: backend unit ≥80%, integration ≥60% по сервисам; роуты — 100% покрыты тестами авторизации.

### Конкретные тесты на SQL-инъекции

1. **Параметризованный builder**: набор тестов, который пытается передать `'; DROP TABLE …` в каждое поле фильтра/сортировки/поиска — должен возвращать пустой результат или 400, никогда 500/успех.
2. **Whitelist полей**: попытка фильтра по полю, не существующему в метамодели сущности — 400 Bad Request.
3. **Rules DSL**: тест, что `LOOKUP('users; DROP TABLE users')` не доходит до SQL (валидируется на уровне AST).
4. **JSONB-запросы**: тесты на `payload->>'field'` с пользовательским вводом — параметризация обязательна.
5. **sqlmap** в staging-пайплайне: автоматический прогон по top-30 endpoint'ов с авторизацией.

### Ключевые связи и инварианты

| Связь | Тип | Инвариант (что обязано выполняться) |
|---|---|---|
| `Application → Module[]` | many-to-many через `application_module` | Все зависимости модуля рекурсивно установлены до самого модуля |
| `Application → ApplicationVersion[]` | one-to-many | Опубликована **только одна** версия в каждый момент; черновик — отдельная активная |
| `Entity → Field[]` | one-to-many | Уникальность `entity.code` + `field.code`; системные поля (`id`, `created_at`, `is_deleted`, `version`) — недоступны для удаления |
| `Field → Rule` | one-to-many, но **1:1 по типу** | Не более одного правила типа `validation` / `autofill` / `visibility` на поле (ТЗ 3.5.2) |
| `Rule → Rule` (зависимости) | граф | Циклы запрещены, проверка при публикации (ТЗ 3.5.3) |
| `Record.entity_id → Entity` | many-to-one | Удаление Entity запрещено при существовании Record (или каскад в корзину) |
| `Record.payload` | JSONB | Схема payload валидируется по `Entity.fields` на каждом write — на чтении доверяем |
| `ProcessInstance → Record` | many-to-one | Один Record может иметь только один **активный** ProcessInstance |
| `AuditLog` | append-only | Триггер запрета UPDATE/DELETE; ежедневная хэш-цепочка `prev_hash → row → hash` |
| `Permission` | (subject, action, resource_pattern, effect) | Permission проверяется на каждом CRUD — в обход PolicyEngine не пройти |
