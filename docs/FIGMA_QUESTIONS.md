## 1. Зачем этот документ

- **Чтобы дизайнер не делал «всё подряд»** в произвольном порядке и не блокировал команду. Команда стартует спринт 2 на неделе 3 — конструктор. Если к нему не будет хотя бы каркаса экранов конструктора в Figma, мы потеряем 1–2 недели.
- **Чтобы зафиксировать UX-решения**, которые архитектура **не предписывает**, но которые **критичны** для безопасности и устойчивости (например: где именно показывать предупреждение о конфликте правил, как визуализировать тайм-аут 113 сек, как пользователь понимает разницу между «черновик» и «опубликовано»).
- **Чтобы избежать перепроектирования** — каждый «вопрос» ниже не «придумать с нуля», а «выбрать из 2–3 вариантов, я ниже подсветил ограничения».

## 2. Приоритеты по спринтам — что нужно когда

Сроки — на конец какого спринта дизайн обязан быть готов к разработке (примерно за 1 неделю до начала спринта BE/FE):

| Что | К концу спринта | Зачем |
|---|---|---|
| Дизайн-система (палитра, типографика, базовые компоненты) | **Спринт 0** | Без неё каждый следующий экран — это переделка |
| Логин, MFA, пароль, профиль | Спринт 0 | Это первое, что увидят тестировщики в спринте 1 |
| Shell приложения (топбар, сайдбар, навигация, переключатель приложений) | Спринт 1 | Каркас всего UI |
| Список приложений + создание (с шаблоном / с нуля) | Спринт 1 | Запуск конструктора в спринте 2 |
| Конструктор: drag-and-drop канвас, добавление блока, edit lock | Спринт 1 | **Самый рисковый экран**, любой пересмотр стоит дорого |
| Редактор сущностей: поля, типы, связи | Спринт 1 | Спринт 2 — моделирование данных |
| Блоки: форма, таблица, кнопка (3 базовых) | Спринт 1 | Без них спринт 2 не запустится |
| Runtime: рендеринг формы/таблицы/страницы | Спринт 2 | Конец спринта 2 — пользователь должен ввести данные |
| Редактор правил: визуальный «ЕСЛИ-ТО-ИНАЧЕ», подсветка конфликтов | Спринт 2 | Спринт 3 — Rules Engine |
| Журнал ошибок правил (для разработчика приложения) | Спринт 2 | Без него Rules — чёрный ящик |
| Конструктор процессов: схема стадий и переходов | Спринт 3 | Спринт 4 — Workflow |
| Моя задача / Согласование (runtime) | Спринт 3 | Спринт 4 — рабочее место исполнителя процесса |
| Корзина + soft delete | Спринт 3 | Спринт 4 |
| Импорт мастер (маппинг полей, отчёт) | Спринт 3 | Спринт 4 |
| Экспорт в Excel/PDF | Спринт 4 | Спринт 5 |
| KPI-блок и диаграммы (3 типа: Bar, Line, Pie) | Спринт 4 | Спринт 5 |
| Регистратор документов | Спринт 4 | Спринт 5 |
| Журнал аудита (для security_officer) | Спринт 4 | Спринт 5 |
| Preview-режим и проверка целостности | Спринт 5 | Спринт 6 |
| База знаний, tooltip, hint, контекстная помощь | Спринт 5 | Спринт 6 |
| Адаптив для runtime (мобильный вид) | Спринт 5 | Спринт 6 |

**Чёрная зона**: блоки 2-й очереди (Kanban, Gantt, Pivot, дерево, расширенные диаграммы), редактор для администратора платформы, мульти-tenant — в MVP не нужны, в дизайне могут отсутствовать.

## 3. Часть A: дизайн-система

### A.1 Что обязательно в дизайн-системе

1. **Палитра**: primary, secondary, semantic (success/warning/error/info), нейтральная серая шкала (10 ступеней). Поддержка тёмной темы — необязательна в MVP, но цвета должны быть совместимы (т.е. не использовать чистые `#000` / `#fff`).
2. **Типографика**: 1 шрифт (Inter / Manrope / системный РФ-friendly), 6–8 размеров, веса 400/500/600. Линейная высота заложена.
3. **Сетка**: 12 колонок desktop, 8 tablet, 4 mobile. Базовый отступ — 4px или 8px (выбрать).
4. **Компоненты-атомы**: Button (4 варианта: primary/secondary/danger/ghost, 3 размера), Input (text/number/textarea/select), Checkbox, Radio, Switch, Date picker, Tag/Chip, Avatar, Icon, Badge, Tooltip, Spinner, Skeleton.
5. **Компоненты-молекулы**: Form Field (label + input + helper + error), Toast, Modal, Drawer, Tabs, Breadcrumbs, Pagination, Empty state, Error state, Loading state.
6. **Компоненты-организмы**: Top bar, Sidebar, Data table (с сортировкой/фильтрами/выбором), Form layout.

### A.2 Вопросы дизайнеру

- ❓ **Иконпак**: Tabler / Lucide / Material? — нужны кириллические иконки (или с поддержкой ru-локали через label). **Не SVG inline, а отдельные компоненты** (мы используем lazy-load и tree-shaking).
- ❓ **Размер сетки** (4 или 8 px) — критично для drag-and-drop в конструкторе.
- ❓ **Брендирование**: будет ли в MVP белый лейбл (возможность кастомизации логотипа/цветов клиентом)? Если да — заложить CSS-переменные.
- ❓ **Иллюстрации** для empty/error states — есть стиль?

## 4. Часть B: аутентификация и онбординг

### B.1 Экраны (обязательны)

1. Логин (email + пароль).
2. Логин через Яндекс ID (кнопка с правильным брендингом — следовать гайдлайнам Яндекса).
3. MFA TOTP (шесть цифр, авто-фокус, autocomplete=`one-time-code`).
4. Восстановление пароля (запрос → email со ссылкой → форма нового пароля).
5. Активация приглашённого пользователя (по ссылке из email — задать пароль + 2FA, если требуется).
6. Профиль (`/me`): аватар, ФИО, email, локаль, таймзона, статус 2FA, список активных сессий.
7. Смена пароля (с проверкой текущего + правилами политики паролей).
8. Включение 2FA: QR-код + ручной ключ + проверка кода.
9. Отзыв сессии (на профиле).

### B.2 Вопросы

- ❓ **Логотип сверху** на экране логина — какой?
- ❓ **Длина пароля и подсказка по правилам** — показывать чек-листом в реальном времени или текстом? (Архитектура: 10+ символов, верх/низ, цифра, спецсимвол.)
- ❓ Дизайн **сообщения «требуется 2FA»** — отдельный экран или попап? (Архитектура: API возвращает 401 + `mfa_session`, FE редиректит.)
- ❓ Что показывать на экране **«ваша сессия истекла»** — модалка с кнопкой «войти заново» или редирект на /login? (Архитектура: сессия 30 мин при бездействии, но незавершённые формы — сохраняются как черновик. UX этого черновика — открыт.)

## 5. Часть C: оболочка приложения (shell)

### C.1 Глобальные элементы UI

1. **Топбар**: логотип, поиск (глобальный), нотификации (звонок), профиль (аватар).
2. **Сайдбар**:
   - Селектор **приложения** (топ — какое приложение я сейчас редактирую/использую).
   - Дерево навигации опубликованного приложения (страницы).
   - Внизу — переключение между режимами: **«Использовать»** и **«Конструктор»** (если есть права).
3. **Хлебные крошки**.
4. **Контекстная панель справа** (drawer) — для деталей записи без перехода на другую страницу.

### C.2 Вопросы

- ❓ **Один сайдбар или два**? (Один с переключателем vs. два — глобальный + контекстный.)
- ❓ **Селектор приложения сверху или в сайдбаре**? (Опыт: в сайдбаре удобнее, в топбаре — заметнее.)
- ❓ Как выглядит **переключатель режимов «Использовать / Конструктор»** — переключатель-toggle или две вкладки наверху?
- ❓ Где **глобальные нотификации**: попап из топбара или отдельная страница `/notifications`?

## 6. Часть D: конструктор приложений

**Самый рисковый экран всей системы.** Это страница, на которой пользователь часами проектирует приложение.

### D.1 Экраны (обязательны)

1. **Список приложений** (моих/всех видимых) — карточки или таблица. С фильтром по статусу (draft/published/archived).
2. **Мастер создания приложения**:
   - Шаг 1: с нуля / из шаблона / из набора модулей.
   - Шаг 2: если модули — каталог с галочками + автоматический расчёт зависимостей.
   - Шаг 3: метаданные (название, иконка, цвет).
   - Шаг 4: подтверждение и установка.
3. **Каркас конструктора** (это **главный экран**):
   - Слева: дерево страниц + дерево сущностей.
   - В центре: канвас выбранной страницы с grid-layout (24-колоночная сетка по умолчанию).
   - Справа: контекстная панель свойств выбранного блока.
   - Сверху: переключатель «Страница / Сущности / Правила / Процессы / Настройки», кнопка «Preview», кнопка «Опубликовать».
4. **Drag-and-drop из палитры блоков** (палитра — drawer слева от канваса или модальный режим).
5. **Edit lock**: «Это приложение редактирует Иванов И. с 14:32. Дождаться / Сообщить» (для admin — «Снять блокировку»).

### D.2 Состояния, которые **обязательно** должны быть в дизайне

| Состояние | Зачем |
|---|---|
| Пустая страница без блоков | Empty state с CTA «Добавить блок» |
| Загрузка | Skeleton, не spinner |
| Несохранённые изменения | Бейдж в топбаре + предупреждение при попытке закрыть |
| Drag-over зона | Чёткая визуальная подсветка зоны вставки |
| Невалидное состояние (например, блок без источника данных) | Иконка-предупреждение на блоке + список ошибок в нижней панели |
| Конфликт правил | Алерт на странице правил + подсветка конфликтующего правила |

### D.3 Вопросы

- ❓ **Сколько колонок на канвасе**: 12 / 24 / гибкая сетка? (12 — простой, 24 — точнее, выбор должен совпадать с тем, что заложит FE-команда в `ui.block.grid_w`.)
- ❓ **Как сворачиваются панели**: persistent vs. collapsible drawer?
- ❓ **Что происходит, когда пользователь нажимает «Опубликовать»** с ошибками целостности? — модалка со списком ошибок или inline-блокировка с указателями на проблемные блоки?
- ❓ **Куда деваются неудалённые черновики версий** — таб «История версий»? Кнопка «Откат» — что в UI?
- ❓ **Клонирование приложения** — куда ведёт после «Клонировать»: к новому или показывает «дубликат создан»?
- ❓ **Где живут общие настройки приложения** (цвет, иконка) — отдельная вкладка «Настройки» или drawer из карточки приложения?

## 7. Часть E: библиотека блоков

Для **каждого** из MVP-блоков нужен **дизайн в трёх состояниях**:

1. **В палитре** (превью + название) — как пользователь его выбирает.
2. **В редакторе** (когда блок добавлен на канвас и выделен) — с правой панелью свойств.
3. **В runtime** (как блок выглядит у конечного пользователя) — в нормальном, error, loading, empty состоянии.

### E.1 Блоки MVP

| Блок | Что важно учесть |
|---|---|
| Текстовое поле | Маска, валидация, helper-текст, error-текст, required-маркер |
| Числовое поле | Сепаратор разрядов, валюта, единица измерения |
| Дата / Период | Single date / date range; календарь с локалью ru |
| Выпадающий список | Single + multi; динамический источник (lookup) с поиском внутри; виртуализация при >100 значений |
| Переключатель (toggle) | Bool, с label слева / справа |
| Загрузка файла | Drag-and-drop зона; список загруженных с прогрессом и av_status |
| Справочник (Lookup) | Selector → modal со списком записей референсной сущности |
| Форма ввода | Layout колонок (1/2/3); группировка полей; required-маркеры; кнопки внизу |
| Таблица | Сортировка, фильтр (по колонке), пагинация (cursor), массовый выбор, контекстное меню по строке, кнопка экспорта |
| Карточка записи | Хедер + поля + связанные записи + история (Activity Timeline) |
| Метрика (KPI) | Большое число + тренд (стрелка, %) + sparkline |
| Диаграмма Bar/Line/Pie | Заголовок, легенда, оси, tooltip, пустое состояние |
| Кнопка | 4 варианта (primary/secondary/danger/ghost) × 3 размера; loading, disabled |
| Блок импорта | Кнопка → мастер импорта (см. §J) |
| Блок экспорта | Кнопка → выбор формата + фильтра |
| Модальное окно | Тригерится из кнопки; вложенные блоки внутри |
| Вкладки | Горизонтальные; адаптив — становятся выпадающим списком на mobile |
| Панель фильтров | Аккордеон с группами условий; сохраняемые пресеты |

### E.2 Вопросы по блоку «Таблица»

Это самый сложный блок. Ниже — открытые вопросы:

- ❓ **Inline-edit** (редактирование прямо в ячейке) — да/нет? (Архитектура поддерживает, UX-стоимость высокая.)
- ❓ **Группировка строк** (по полю) — в MVP или 2-й очереди?
- ❓ **Закреплённые колонки** слева/справа — да/нет?
- ❓ **Бесконечная прокрутка vs. явная пагинация** — выбрать одно для MVP.
- ❓ Где **массовые действия** появляются (после выбора нескольких строк) — sticky-тулбар сверху или контекстное меню?
- ❓ Как **показать удалённые записи** (`?include_deleted=true`) — отдельная вкладка «Корзина» или toggle над таблицей?

### E.3 Вопросы по блоку «Форма»

- ❓ **Layout колонок** — фиксированный (1/2/3) или free-form (drag-and-drop полей внутри формы)?
- ❓ **Условная видимость поля** (визибилити-правило) — как пользователь это настраивает в редакторе формы?
- ❓ **Группировка полей** в секции с заголовками — поддерживается?
- ❓ **Сохранение черновика автоматом** при потере фокуса формы — заложено в API через идемпотентность; UX-индикатор «черновик сохранён» — нужен?

## 8. Часть F: редактор сущностей

### F.1 Экраны

1. **Список сущностей приложения** (таблица: код, имя, кол-во записей, активность).
2. **Карточка сущности**:
   - Вкладка «Поля» — таблица полей с inline-edit (тип, required, indexed, PII).
   - Вкладка «Связи» — диаграмма (мини-ER) + список.
   - Вкладка «Валидаторы».
   - Вкладка «Политики хранения и удаления».
   - Вкладка «Записи» (быстрый просмотр данных).
3. **Мастер добавления поля**:
   - Шаг 1: тип поля (text/number/decimal/date/datetime/boolean/enum/file/relation/formula/currency).
   - Шаг 2: параметры (для каждого типа — свой набор).
   - Шаг 3: валидаторы, флаги (required, indexed, PII).

### F.2 Вопросы

- ❓ **Удаление поля с данными** — модалка-предупреждение или вообще запрет? (Архитектура: данные в `record.payload` JSONB остаются, но недоступны через UI; миграция «удалить из payload» — отдельная админ-операция.)
- ❓ **Изменение типа поля** (например, text → number) — что показываем? Архитектурно это очень опасно: что делать с несовместимыми данными? Предлагаю: **запретить в MVP**, поле удаляется и создаётся новое с новым типом.
- ❓ **Формула-поле**: визуальный редактор формулы (тот же, что и для правил)? Где он живёт — drawer или отдельная страница?

## 9. Часть G: редактор правил

**Сложность по UX**: рядом с конструктором приложений. Это, фактически, no-code язык программирования с визуальным интерфейсом.

### G.1 Экраны

1. **Список правил приложения** — таблица: код, имя, тип (валидация/автозаполнение/...), сущность, поле, приоритет, статус (active/draft/paused), вкл/выкл-переключатель.
2. **Редактор правила** — главный экран:
   - Метаданные сверху: код, имя, тип, приоритет, тайм-аут.
   - Секция **ЕСЛИ** — визуальный builder условия (см. ниже).
   - Секция **ТО** — список действий.
   - Секция **ИНАЧЕ** (опционально) — список действий.
3. **Журнал выполнения** правила (per-rule view).
4. **Журнал конфликтов** (один на приложение).
5. **Тест правила** — модалка с заполненной фиктивной записью, кнопка «Прогнать», результат.

### G.2 Визуальный builder условия — главная UX-задача

Условие в архитектуре — это JSON-AST с операторами `eq/gt/and/or/lookup/...` (см. `RULES_AND_WORKFLOW.md` §2). Дизайнеру нужно решить:

- **Стиль интерфейса**:
  - Вариант A — «строки условий» (как в Salesforce / Bubble): один ряд = одно сравнение, кнопка «+ И / + ИЛИ» добавляет вложенный блок.
  - Вариант B — «древовидный builder» (как в react-querybuilder): вложенные AND/OR группы с явным выравниванием.
  - Вариант C — «формула-строка» (как в Airtable): свободный текст с автодополнением.
  - **Архитектура поддержит любой**, но мы рекомендуем **B (древовидный)** — даёт максимально ясную ментальную модель «вложенных условий», которую требует ТЗ 3.5.2.

- **Drag-and-drop узлов** — нужен или фиксированная структура?
- **Подсветка типов** — в селекторе поля сразу показываем тип (`amount [число]`), при выборе оператора — отфильтрованный список под этот тип.
- **`lookup` как отдельный «бокс»** — визуально выделяется, потому что он медленный (см. §G.5).

### G.3 Действия (`ТО`) — UX

Каждое действие — карточка с типом (`set_field` / `notify` / `block_save` / `create_record` / `webhook`) и параметрами. Перетаскивание — изменение порядка.

### G.4 Конфликты правил — как визуализируем

ТЗ 3.5.4: «Редактор визуально предупреждает о потенциальных конфликтах до публикации». Варианты:
- A: бейдж на правиле в списке.
- B: модалка при сохранении: «Это правило конфликтует с R5».
- C: отдельная вкладка «Конфликты».
- Рекомендация: **A + C** (бейдж + диагностика в спец-вкладке).

### G.5 Видимость стоимости

`lookup` — дорогой. Дизайн должен показать пользователю, что условие «дорогое»:
- Бейдж «⚡ медленно» рядом с узлом `lookup` если >1 вызова.
- В журнале выполнения — отдельная цифра «вызовов lookup: 4».

### G.6 Вопросы

- ❓ Выбор стиля **builder'а** (см. G.2).
- ❓ Что показываем, когда правило **в circuit-breaker** (3 таймаута подряд → деактивировано) — модалка с предложением «исправить и активировать»?
- ❓ Как пользователь **тестирует правило**: модалка / отдельный экран / inline-preview?
- ❓ Куда выводим **граф зависимостей правил** (для отладки циклов) — отдельная вкладка с визуализацией или просто список?

## 12. Часть J: данные — корзина, импорт, экспорт

### J.1 Корзина

- Единый экран `/recycle-bin` (приложение) — список soft-deleted записей по всем сущностям.
- Также — таб «Корзина» в карточке сущности.
- Действия: «Восстановить», «Удалить навсегда» (только `record.purge` — отдельное право DPO).
- Показать срок до автоархивации (по retention_policy).

### J.2 Импорт — мастер

Шаги:
1. Выбор сущности.
2. Загрузка файла (drag-and-drop, .xlsx/.csv/.json).
3. Маппинг колонок → полей сущности (auto-detect + ручная коррекция).
4. Превью первых 10 строк с подсветкой ошибок валидации.
5. Выбор режима (skip errors / abort on error / update by key).
6. Запуск (асинхронно).
7. Прогресс + отчёт по завершении.

### J.3 Экспорт — диалог

- Формат: XLSX / CSV / PDF / JSON.
- Колонки: выбор из доступных (с учётом маскирования).
- Фильтр: текущий из таблицы или построитель.
- Опция «применять маскирование PII» (по умолчанию — да).
- Запуск (асинхронно для >1000 строк), уведомление по готовности.

### J.4 Вопросы

- ❓ **Маппинг колонок** — auto-detect эвристиками (имя колонки = код поля) или полностью ручной? (Архитектурно поддерживается оба, выбор UX.)
- ❓ Что показываем в **отчёте об ошибках импорта** — список первых 100 ошибок + CSV-выгрузка всех?
- ❓ Где **журнал импортов/экспортов** (история, кто, когда, сколько) — отдельный экран `/jobs`?

## 18. Открытые UX-решения, где нужен голос дизайнера

Это вопросы, в которых **архитектура не диктует** — решает дизайн:

1. **Что увидит data_user при попытке открыть приложение, на которое нет прав**: 404 / 403 / редирект на список доступных приложений?
2. **Состояние сети потеряно** (offline) — banner / toast / модалка?
3. **Сохранение незавершённой формы при истечении сессии** — где живёт черновик и как пользователь его восстанавливает?
4. **Удаление записи с зависимостями** (есть связанные записи): модалка-предупреждение vs. блокировка vs. каскад с подтверждением?
5. **Изменение схемы сущности** при существующих записях — каким UX подаём такой риск пользователю?
6. **Редактор условий ABAC**: как объяснить пользователю-нетехническому, что такое «свой отдел»? (Шаблоны политик: «свои записи», «свой отдел», «свои подчинённые».)
7. **Препрод-режим** (preview опубликованного приложения перед публикацией) — как пользователь понимает, что он в preview, а не в реальном приложении? (Цветной баннер?)
8. **«Опубликовать»** — куда возвращаем пользователя после успеха? К списку приложений или остаёмся в конструкторе с пометкой «опубликовано»?
9. **Бейдж «версия N»** — отображается всегда или только при не-последней?
10. **Антивирус (av_status=pending)** — как показываем «файл загружен, проверка...» — кнопка «Скачать» неактивна с тултипом?
